# 核心教學閱讀筆記

本文件是對已讀核心帖的原創整理，不複製論壇全文。

## 新手入門：Thread 6664

來源：<https://bbs.zdjl.site/#/thread/detail/6664>

重點理解：

- 自動精靈不是單純連點器，核心差異在「變量 + 條件 + 識別 + JS」。
- 新手最容易忽略全局功能，例如重複次數、出錯暫停、全局監聽。
- OCR 可透過「識別屏幕」或「設定變量」保存結果。
- 顏色識別在特徵明確時比圖片識別快。
- 濾鏡可提高 OCR 或圖像識別成功率，例如去掉淡色副標題。
- 同時運行多個動作可提升效率，但要注意競態和畫面狀態。
- 本地儲存適合批量保存 OCR、帳號、任務結果等資料。
- 正則表達式適合從 OCR 或網路返回文字中提取數字、連結、時間。

對腳本架構的啟發：

- 每個模組要有日誌或提示，不然後期很難知道卡在哪一段。
- 新手也能先用積木封裝「能穩定做的動作」，再用 JS 處理邏輯。

## 基礎教程 2.0：Thread 11246

來源：<https://bbs.zdjl.site/#/thread/detail/11246>

重點理解：

- 這篇偏 JavaScript 基礎與自動精靈官方函數整理。
- 內容涵蓋資料型別、運算符、條件、邏輯、日期、數學、布林、數字、字串、陣列、通用函數。
- 對自動精靈來說，JS 基礎不是為了寫大型程式，而是為了能穩定處理表達式。

實戰用途：

- 日期與時間：定時任務、跨天判斷、倒計時。
- Math：隨機資源、隨機點擊、坐標偏移、防固定行為。
- String：OCR 結果清洗、角色名比對、提取數字。
- Array：角色列表、資源隊列、任務隊列。
- Boolean：運行條件與狀態標記。

## 變量封裝與 AI 輔助：Thread 25429

來源：<https://bbs.zdjl.site/#/thread/detail/25429>

重點理解：

- 大型腳本最容易亂在「A 失敗跳 B，B 失敗跳 C」這種跳轉鏈。
- 更好的做法是先把動作封裝成變量或動作組，再用 JS 負責調度。
- AI 適合幫忙整理流程圖、寫判斷邏輯、處理表達式。
- 動作內容可以透過查看變量表達式提供給 AI 分析。

對萬龍覺醒腳本的應用：

- `V1-G 登入角色`、`V1-C 採集派遣`、`V1-D 收尾` 都應該是可獨立調用的模組。
- 圖色/OCR/點擊這類動作先封裝好，再由 JS 根據狀態決定調哪一段。
- 所有失敗分支要有統一出口，而不是無限跳。

## 變量做任務循環：Thread 14838

來源：<https://bbs.zdjl.site/#/thread/detail/14838>

重點理解：

- 可用變量保存「一組動作」，再透過運行多個動作調用。
- 任務末尾再調用自己，就能形成可控循環。
- 循環停止條件要放在運行條件裡，條件成立就停止，不成立才繼續。

設計注意：

- 循環必須有停止條件。
- 重複次數、等待時間、失敗處理要明確。
- 適合做單任務反覆嘗試，例如採集搜尋、等待任務完成、反覆關閉彈窗。

## 外部 OCR：Thread 25439

來源：<https://bbs.zdjl.site/#/thread/detail/25439>

重點理解：

- 當本地 OCR 不準時，可以用外部 OCR API 作為替代方案。
- 基本流程是：截圖指定區域，轉 base64，發 POST，解析返回文字。
- 實作時要把 API key、圖片區域、結果保存、錯誤日誌分開處理。
- 可先在 PC 跑通，再改成手機/雲手機可用路徑。

安全與穩定建議：

- API key 不要放進公開 GitHub。
- 外部 OCR 有額度、延遲、網路失敗問題，不適合高頻循環。
- 遊戲腳本中應優先縮小截圖區域，減少識別錯誤與傳輸成本。
- 能本地識別就本地識別，外部 OCR 作為備用方案。

## 多點找色：Thread 16545

來源：<https://bbs.zdjl.site/#/thread/detail/16545>

重點理解：

- 單點找色容易誤判，因為同色可能到處都有。
- 多點找色是用「基準點 + 相對位置」建立圖案特徵。
- 找到基準色後，再檢查附近多個相對位置是否符合，符合才點擊。

遊戲應用：

- 判斷固定 UI 圖標。
- 判斷按鈕是否存在。
- 判斷資源/技能/狀態圖示。
- 比單張模板找圖更輕量，但需要顏色穩定。

## UI、局數、運行時長：Thread 25398

來源：<https://bbs.zdjl.site/#/thread/detail/25398>

重點理解：

- UI 可以顯示局數、運行時長、當前時間、提示文字與顏色。
- `Date.now()` 與腳本啟動時間可計算運行時長。
- Markdown 風格提示能讓提示更清楚。

腳本應用：

- 萬龍覺醒腳本可顯示目前角色、目前任務、部隊數、資源類型、錯誤原因。
- 長時間腳本要有狀態面板，否則只靠截圖很難追查。

## 分頁 UI：Thread 16319

來源：<https://bbs.zdjl.site/#/thread/detail/16319>

重點理解：

- 分頁 UI 的核心是多個布林變量控制不同區塊顯示/隱藏。
- 按鈕點擊後執行 JS，切換哪個布林為真。
- 每個頁面的變量顯示屬性由對應布林控制。

應用：

- 可做腳本設定頁，例如角色設定、採集設定、OCR 設定、日誌設定。
- 對多帳號腳本很有用，可以把配置拆頁而不是堆在一個畫面。

## 血條/進度條百分比：Thread 18438

來源：<https://bbs.zdjl.site/#/thread/detail/18438>

重點理解：

- 進度條判斷可透過區域顏色資料換算百分比。
- 比 OCR 倒計時更快，也更適合判斷血條、冷卻、採集進度。
- 前提是條形或圓形進度的顏色對比足夠明顯。

應用：

- 判斷資源採集中進度。
- 判斷隊伍狀態條。
- 判斷技能冷卻或活動進度。

## 查看動作表達式與源碼：Thread 19076

來源：<https://bbs.zdjl.site/#/thread/detail/19076>

重點理解：

- 自動精靈動作可轉成結構化資料，JS 可以動態建立並執行。
- `runAction` 和 `runActionAsync` 是把積木變成代碼調度的關鍵。
- `check` 和 `checkAsync` 可檢查運行條件是否成立。
- 遠程編輯或查看動作源碼，有助於理解 `.zjs` 內部結構。

對修改 `.zjs` 的意義：

- 看到動作 JSON 後，可以理解每個積木的 type、posData、condition、scriptCallbacks。
- 修大型腳本時不要只改顯示名稱，要看實際動作資料是否一致。
- 可以把重複動作抽成可調用模板。
