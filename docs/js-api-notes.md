# 自動精靈 JS/API 學習筆記

來源主帖：<https://bbs.zdjl.site/#/thread/detail/1380>

## 角色定位

「运行 JS 代码」是自動精靈裡最靈活的擴展方式。它不是取代積木，而是用來補足積木難以表達的部分，例如：

- 複雜條件判斷
- 字串處理
- 陣列/清單管理
- 動態座標
- OCR 結果解析
- 網路請求
- 文件讀寫
- 批量執行動作

## 通用能力

官方文檔列出的通用能力可整理成幾類：

- App 與系統：版本號、剪貼板、睡眠等待。
- 提示互動：toast、alert、confirm、prompt、select。
- Markdown 提示：新版支援用 Markdown 顯示 richer 提示。
- 日誌：`console` 面板可用來顯示調試、警告、錯誤。

理解方式：

- `toast` 適合流程中短提示。
- `alert/confirm/prompt/select` 適合人工介入或測試設定。
- `console` 適合長期除錯，不應被一般提示取代。

## 變量讀寫

JS 可以直接讀寫「設定變量」動作建立的變量，也可以透過 `zdjl.getVar` 和 `zdjl.setVar` 操作。

重點：

- 當前腳本作用域和全局作用域要分清楚。
- 直接給已存在變量賦值，效果接近重新設定變量。
- 可以刪除單一變量，也可以清理當前腳本或全局變量。
- `printVars` 適合調試，正式腳本不建議長期保留。

實戰建議：

- 角色索引、角色名陣列、採集資源隊列屬於跨段資料。
- OCR 原始結果、單次搜尋結果、單次座標屬於臨時資料。

## 動作調用

JS 可動態建立並運行動作，分同步與異步兩種思路：

- 同步運行：適合簡單、立即完成的動作。
- 異步運行：適合需要等待結果或串接流程的動作。

這讓腳本可以把「重複但參數不同」的動作封裝成函數，例如：

- 點擊百分比座標
- 統一返回主畫面
- 根據 OCR 座標點角色
- 根據資源類型點不同按鈕

## 找座標

`findLocation` 的核心用途是把「條件」轉成「可點擊座標」。

常見條件：

- 文字
- 節點
- 圖片
- 顏色/多色

遊戲腳本注意：

- 文字 OCR 容易被背景文字干擾，必須限制區域。
- 找角色名時只掃角色管理列表，不掃世界地圖。
- 找按鈕時優先限制在彈窗或面板區域。

## 找節點

`findNode` 適合普通 App 或系統界面。它能取得文字、類名、包名、邊界等資訊。

遊戲自繪畫面通常節點少，不能把它當主要方案；但在雲手機、啟動器、權限彈窗、自動精靈本身 UI 裡仍然有用。

## 屏幕識別

`recognitionScreen` 可把「變量-屏幕識別」能力搬到 JS 裡使用。

官方文檔提到的模式包含：

- 本地 OCR
- 本地 OCR 兼容模式
- 雲 OCR
- 圖像驗證碼
- 取得圖片資料
- 人工坐標識別

實戰建議：

- 遊戲角色名、資源提示、倒計時適合 OCR。
- 血條/進度條不一定適合 OCR，找色或區域顏色更穩。
- OCR 結果要做清洗，例如去空格、大小寫統一、只保留數字。

## 顏色與區域顏色

可取得單點顏色，也可取得區域顏色陣列。

適合：

- 判斷按鈕是否亮起
- 判斷技能冷卻
- 判斷進度條比例
- 判斷畫面是否卡住

性能上，顏色判斷通常比 OCR 和找圖快，適合高頻循環。

## OCR 直接識別圖片

官方文檔也提到可對 base64 圖片做 OCR，並支援原始結果。

原始結果重要性：

- 可以取得文字片段和座標。
- 可用來做「找到角色名後點該文字中心」。
- 可用來避開固定座標方案。

## 網路請求

自動精靈 JS 可做網路請求，適合：

- 查時間
- 拉取遠端配置
- 熱更新
- 上報狀態
- 驗證卡密
- 通知完成

安全建議：

- 不要把私密 token 寫死在公開 GitHub。
- 遠端返回內容要做格式檢查。
- 網路失敗要有降級方案，不能讓腳本死循環。

## 學習結論

自動精靈的強點不是單一點擊，而是「積木流程 + 變量 + JS + OCR/找圖/找色」混合架構。大型腳本要把每個動作做成可判斷、可重試、可退出的狀態機。
