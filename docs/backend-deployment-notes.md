## 后端能力要解决什么

自动精灵脚本需要后端时，通常不是为了“看起来高级”，而是为了处理本地脚本做不到或不适合做的事情：

- 用户管理：登录、设备绑定、黑白名单、到期时间。
- 卡密验证：兑换、续期、封禁、查询。
- 热更新：检测版本、下载新脚本、强制更新、灰度更新。
- 配置下发：资源等级、任务开关、公告、禁用列表。
- 远程 OCR：本地识别不准时，把截图发给 PC 或云服务。
- 日志回传：错误统计、运行状态、异常设备。
- 动态内容：留言、验证码、排行榜、任务数据。

## 技术路线对比

| 路线 | 适合场景 | 优点 | 风险 |
|---|---|---|---|
| GitHub / 静态托管 | 静态配置、脚本下载、说明文档 | 简单、免费、适合公开文件 | 不能执行 PHP，不能做动态登录或数据库 |
| PHP + MySQL 托管 | 卡密、验证码、简单 API、配置读写 | 文件即服务，部署轻，适合小白和 AI 改代码 | 免费平台稳定性不一，要保护后台和数据库 |
| Supabase | 用户系统、数据库、认证、存储 | 免自建服务器，管理界面完善，适合现代后端 | 学习成本高，权限策略和密钥管理要小心 |
| Node.js + Express + MongoDB | 完整自建后端、复杂业务 | 控制力强，接口结构清楚，适合长期项目 | 要会服务器、端口、进程、数据库和部署 |
| 第三方热更新/卡密平台 | 快速上线、少写后端 | 省时间，有后台 | 平台跑路、接口变化、隐私和可控性风险 |
| 局域网 PC OCR | 高频 OCR、复杂识别 | 速度快，可利用 PC 算力 | 要同网段/公网映射，防火墙和稳定性要处理 |

## 热更新架构

来自：`25309`, `21693`, `17671`

热更新要分成三层：

1. 版本信息层：服务器保存当前版本、最低可用版本、更新类型和下载地址。
2. 启动器层：脚本启动时检查版本、决定是否更新、下载并覆盖业务脚本。
3. 业务层：真正执行自动化任务，不关心更新细节。

重要经验：

- 启动器和业务脚本分离，比把更新逻辑塞进每个旧脚本更好维护。
- 版本比较不能用普通数字比较，例如 `3.8` 和 `3.11` 会出错，应做版本号分段比较。
- 远程版本请求可能被缓存，关键更新检查可改用 `POST` 或加时间戳。
- 更新类型至少要区分：无需更新、可选更新、强制更新。
- 覆盖写入脚本前要先验证下载内容不是空、不是错误页、不是旧版本。

GitHub 公开时只放：

- 更新流程图。
- 脱敏字段名，例如 `version`, `minVersion`, `downloadUrl`, `forceUpdate`。
- 伪代码或简化模板。

不要放：

- 真实后台地址。
- 管理 token。
- 完整商业热更新脚本。

### 热更新 manifest 建议

公开 GitHub 可以放字段结构，但不要放真实下载地址或后台 token。

```json
{
  "appId": "wanlong-gather",
  "channel": "stable",
  "latestVersion": "1.2.0",
  "minVersion": "1.0.0",
  "forceUpdate": false,
  "downloadUrl": "https://example.invalid/releases/wanlong-gather.zjs",
  "sha256": "replace-with-fake-hash",
  "message": "修复资源搜索失败恢复逻辑",
  "publishedAt": "2026-06-15T00:00:00+08:00"
}
```

脚本端检查顺序：

1. 请求 manifest。
2. 确认 JSON 可解析。
3. 比较 `minVersion`，低于最低版本则停止或强制更新。
4. 比较 `latestVersion`，决定是否提示。
5. 下载内容前检查 `downloadUrl` 是否来自允许域名。
6. 下载后检查大小、hash、脚本头部标识。
7. 写入前备份旧版本。
8. 写入失败时回滚旧版本。

### 版本号比较

不要用浮点数比较版本。`3.11` 转成数字会小于 `3.8`，这是热更新常见坑。

建议把版本切成数组比较：

```js
function compareVersion(a, b) {
  const aa = String(a).split('.').map(Number);
  const bb = String(b).split('.').map(Number);
  const len = Math.max(aa.length, bb.length);
  for (let i = 0; i < len; i++) {
    const x = aa[i] || 0;
    const y = bb[i] || 0;
    if (x > y) return 1;
    if (x < y) return -1;
  }
  return 0;
}
```

## 用户管理与卡密

来自：`25311`, `20500`, `5068`

卡密系统最少需要这些字段：

- 用户或设备标识。
- 卡密或兑换码。
- 到期时间。
- 状态：可用、已用、封禁、过期。
- 绑定策略：一机一码、多设备、IP 辅助、账号登录。

可靠性排序：

1. 自建后端或 Supabase：最可控。
2. 第三方卡密平台：上线快，但依赖平台。
3. 静态白名单：简单但维护麻烦。
4. 本地加密卡密：适合学习，不适合严肃商业授权。

安全注意：

- 不要把验证算法和密钥都放在客户端公开脚本里。
- 客户端只能做初步检查，最终有效性最好由服务端判断。
- 设备 ID、IP、用户 ID 都不是绝对可靠，只能作为组合风控信号。
- 公开 GitHub 只能展示脱敏流程，不展示可直接滥用的验证接口。

### 授权请求的安全边界

脚本端可以提交：

- 脱敏设备指纹。
- 用户输入的兑换码。
- 当前脚本版本。
- 随机 nonce。
- 时间戳。

脚本端不应该保存：

- 高权限服务端密钥。
- 数据库密码。
- 后台管理 token。
- 卡密生成算法。
- 私人接口完整地址。

服务端应该返回：

```json
{
  "ok": true,
  "code": "ACTIVE",
  "message": "authorized",
  "data": {
    "expiresAt": "2026-12-31T23:59:59+08:00",
    "features": ["gather", "multi_role"],
    "serverTime": "2026-06-15T00:00:00+08:00"
  }
}
```

脚本端只根据 `ok/code/expiresAt/features` 做流程分支，不要把服务端返回的文本直接 `eval` 或当脚本执行。

### 第三方登录

第三方登录教程可以学习「开放接口 + 回调 + 用户身份」的架构，但 GitHub 学习库只记录概念：

- 登录用于确认用户身份，不等于授权全部功能。
- 登录成功后仍应由后端判断功能权限。
- OAuth、QQ 登录等流程不要把 app secret 写入客户端脚本。
- 回调地址、app id、app secret 都要脱敏。

## PHP/MySQL 轻量后端

来自：`13820`, `13854`, `22321`, `22209`

PHP 后端适合自动精灵的原因：

- 不需要手动常驻运行进程，请求 PHP 文件就会执行。
- 改文件即可更新接口，部署轻。
- 免费虚拟主机或 WordPress 类托管也能跑。
- 简单接口用原生 PHP + MySQL 就够。

常见接口形态：

- `POST /save.php`：保存文本、配置、日志。
- `GET /get.php?id=...`：读取配置或文本。
- `GET /captcha.php`：生成验证码。
- `POST /validate.php`：核销验证码或卡密。

必须注意：

- 使用预处理语句，避免 SQL 注入。
- 返回 JSON，脚本端统一按 `status/message/data` 解析。
- 每个接口都要处理参数缺失、数据库失败、重复提交。
- 免费托管平台可能换域名、限流或限制文件操作，脚本端要能提示错误。

### API 回应格式

无论 PHP、Node 还是 Supabase Edge Function，都建议统一返回：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {},
  "serverTime": "2026-06-15T00:00:00+08:00",
  "requestId": "example-request-id"
}
```

错误也返回 JSON：

```json
{
  "ok": false,
  "code": "MISSING_PARAM",
  "message": "missing required field",
  "data": null,
  "serverTime": "2026-06-15T00:00:00+08:00",
  "requestId": "example-request-id"
}
```

脚本端解析时：

- 先判断网络请求是否成功。
- 再判断 body 是否能 JSON.parse。
- 再判断 `ok`。
- 最后才读取 `data`。
- 出错时写入日志与状态，不要直接卡死。

## Node.js + Express + MongoDB

来自：`19365`

适合有编程基础、要长期维护的脚本作者。基本链路：

1. 安装 Node.js，并确认 `node -v`、`npm -v` 可用。
2. 安装数据库，教程使用 MongoDB，因为 JSON 数据比较自然。
3. 用 Express 写 API。
4. 脚本端通过网络请求发送数据。
5. 服务端写入数据库。
6. 脚本端按条件查询服务器数据。

实践注意：

- 路径尽量不要有中文，减少环境问题。
- 局域网使用只需要同 WiFi；跨网络需要云服务器、公网 IP 或内网穿透。
- 服务器要有日志、错误处理、参数校验。
- 如果只是简单卡密和配置，不一定要上 Node 全家桶，PHP 或 Supabase 可能更轻。

### Node 路线最小模块

若真的要自建 Node 后端，建议只从最小模块开始：

| 模块 | API | 说明 |
|---|---|---|
| health | `GET /health` | 检查服务是否活着 |
| version | `GET /version/:appId` | 返回热更新 manifest |
| auth | `POST /auth/verify` | 验证用户、设备、到期时间 |
| config | `GET /config/:profile` | 下发脱敏配置 |
| log | `POST /log` | 上报错误摘要 |

每个 API 都要有：

- 参数校验。
- JSON 结果。
- 错误码。
- 请求日志。
- 速率限制。
- 不泄漏堆栈信息。

## Supabase 路线

来自：`25311`

Supabase 更像是“现成后端平台”，可提供：

- PostgreSQL 数据库。
- 用户认证。
- 文件存储。
- 实时能力。
- 管理界面。
- 与 AI Agent 配合生成后端结构。

适合：

- 要做用户管理。
- 要保存多张表。
- 要图形界面管理数据。
- 不想自己维护服务器进程。

风险：

- 需要理解权限策略，尤其是公开 key 和服务端 key 的区别。
- 不要把高权限密钥写入脚本或 GitHub。
- 脚本端 SDK 要适配自动精灵 JS 环境，不能直接假设浏览器或 Node 完整环境。

### Supabase 密钥分层

Supabase 常见误区是把高权限 key 放进客户端。整理规则：

| key 类型 | 能否放脚本端 | 说明 |
|---|---|---|
| anon/public key | 可在规则正确时使用 | 仍要配 RLS 权限 |
| service role key | 不可 | 只能放后端或可信服务器 |
| 数据库连接串 | 不可 | 不能出现在公开仓库 |
| JWT secret | 不可 | 泄漏后可伪造认证 |

GitHub 范例只写：

```json
{
  "supabaseUrl": "https://example.supabase.co",
  "supabaseAnonKey": "replace-with-public-anon-key"
}
```

并明确说明真实项目要检查 RLS 规则。

## 远程 OCR 与 PC OCR

来自：`25093`

PC OCR 适合本地 OCR 不准或需要高频识别的场景。基本链路：

1. PC 启动 Umi-OCR 或 Paddle OCR 服务。
2. 手机和 PC 在同一局域网，或 PC 暴露公网地址。
3. 脚本截取指定区域图片。
4. 脚本把图片发给 PC OCR 接口。
5. 解析返回文字或坐标。

注意：

- PC 防火墙和端口必须放行。
- 脚本端要配置 PC IP 和端口。
- 外网访问要用公网 IP 或内网穿透。
- 远程 OCR 有延迟，不适合每帧高频调用。
- 小图片识别能力要实测，不同 OCR 引擎差异很大。

## 配置持久化

来自：`23640`

脚本本地配置可以分三层：

- 默认配置：写在脚本里，首次启动使用。
- 用户配置：通过 UI 或变量输入。
- 持久配置：通过本地存储保存，下次运行读取。

适合保存：

- 角色列表。
- 资源类型。
- 采集等级。
- OCR 区域。
- 是否启用某模块。
- 多套游戏/账号配置。

建议：

- 配置名称列表单独保存。
- 每套配置用独立 key，例如 `config:角色1`、`config:木材队列`。
- 保存前校验字段，读取失败时回退默认值。

### 本地优先原则

万龙采集脚本第一版建议本地优先：

1. UI 输入配置。
2. 保存到 storage / JSON。
3. 运行时读取本地配置。
4. 后端只作为后续发布、授权、远程配置的扩展。

这样可以减少早期变量：

- 网络不可用。
- 后端接口改动。
- 免费服务限流。
- 授权误判。
- GitHub 公开时不小心泄漏接口。

## IP 与风控

来自：`21819`

IP 和归属地可用于：

- 黑名单辅助。
- 登录风险提示。
- 留言归属地展示。
- 同 IP 多设备统计。

限制：

- VPN、代理和运营商出口会影响准确性。
- 只能作为辅助信号，不应当作为唯一授权判断。
- 公开脚本中要写清楚隐私用途，避免过度收集。

## GitHub 收录规则

可以收录：

- 架构图。
- 接口字段说明。
- 脱敏请求示例。
- 本地配置结构。
- 安全注意事项。
- 失败处理策略。

不要收录：

- 真实卡密接口。
- 后台管理地址。
- API 密钥。
- 数据库账号密码。
- 商业脚本完整授权逻辑。
- 可直接绕过或滥用第三方服务的抓包细节。

## 发布前安全检查

后端相关内容发布前必须检查：

```powershell
rg "http://|https://|token|secret|password|passwd|pwd|key|密钥|密鑰|卡密|后台|後台|管理地址" README.md docs sources prompts examples tests
```

命中后逐条确认：

- 是否只是安全提醒。
- 是否是假域名，例如 `example.invalid`。
- 是否是真实后台地址。
- 是否包含真实 key、token、数据库账号。
- 是否包含可直接复制使用的卡密验证逻辑。

真实项目建议把敏感配置放在：

- 后端环境变量。
- 私有配置文件。
- 云平台 secret manager。
- 不进入 GitHub 的本地 `.env`。

## 对萬龍覺醒脚本的应用

- 采集脚本可以先以本地配置为主，不急着上后端。
- 需要多账号、多角色、多资源队列时，先做本地配置持久化。
- 若要发布给别人使用，再考虑热更新和版本检测。
- OCR 不稳定时，可预留 PC OCR 地址配置，但默认不用远程 OCR。
- GitHub 公开版只放脱敏模板和架构，不放实际授权、卡密、后端密钥。

### 万龙后端演进路线

| 阶段 | 做什么 | 不做什么 |
|---|---|---|
| 1. 本地脚本 | 本地配置、状态文件、错误截图 | 不接卡密，不强依赖后端 |
