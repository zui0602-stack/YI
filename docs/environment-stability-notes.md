## 無障礙與權限

Android 13 以後常見「受限制的設置」或無障礙開關變灰。已讀內容整理出的公開做法：

- 從系統的應用詳情頁進入，允許「受限制的設置」後再開無障礙。
- 若安裝來源導致權限頁看不到自動精靈，可嘗試用可信文件管理器或系統允許的安裝方式重新安裝。
- 系統更新後權限可能被重置，腳本測試前要把無障礙、懸浮窗、截屏錄製、後台運行都列入檢查表。
- 不同品牌限制不同，小米、真我、iQOO、一加、聯想平板都出現過個案；問題記錄要包含機型、Android 版本、自動精靈版本與安裝來源。

不整理、不收錄的內容：

- 繞過應用偵測無障礙的做法。
- 破解限制、隱藏服務、繞過平台安全機制的做法。
- 任何可能讓 GitHub 倉庫變成濫用工具的完整流程。

## root、無障礙與運行模式

分類 4 多個案例顯示 root 模式、無障礙模式、暫停/停止機制之間可能互相影響。公開筆記採用保守結論：

- 不要假設 root 和無障礙可以無縫混用。
- 腳本開發階段先選定一個主要運行模式，測穩後再做另一套環境。
- root 後無法暫停、停止不了、只能重啟這類問題，要視為高風險環境，不能拿來當第一測試平台。
- 長時間腳本必須做外部停止保護，例如音量鍵停止、UI 停止旗標、定時退出或雲機外部控制。

## 模擬器與雲手機

真機、模擬器、紅手指雲手機即使解析度一樣，也不代表坐標和截圖資料一樣。差異可能來自：

- DPI 與縮放比例。
- 劉海、導航列、狀態列。
- 遊戲渲染比例。
- 模擬器 32 位/64 位。
- 顯卡渲染模式。
- 錄屏與截圖權限。

常見處理：

- 優先使用 64 位模擬器與足夠性能配置。
- OCR 或屏幕識別報錯時，先把模擬器內存提高到 4GB 以上，再測兼容模式。
- 半屏識別可嘗試旋轉屏幕兩次，重新建立截圖方向。
- 識圖全黑時檢查是否彈出過截屏/錄屏授權；必要時重啟自動精靈讓授權彈窗重新出現。
- 多開環境要降低操作頻率，增加等待與重試，因為每個實例都會拖慢識別與點擊。
- 雲手機重啟、失聯或系統殺後台，不一定能靠腳本內自救，應另做外部監控。

對紅手指測試：

- 每台雲機先建立一份「設備檔案」：解析度、DPI、Android 版本、是否有導航列、OCR 模式、截圖是否正常。
- 腳本啟動前先跑一個 10 秒自檢：截圖、OCR 小區域、找圖、點擊回城/放大鏡區域、停止鍵。
- 自檢失敗就不要進萬龍採集主流程，先保存截圖與日誌。

## OCR ERROR 與文字識別卡死

已讀交流帖反覆出現同一類結論：OCR 不能高頻、不能無限制全屏掃，也不適合多個動作同時檢測。

推薦規則：

- OCR 必須限制區域，只掃需要的文字。
- 主循環中 OCR 加冷卻時間，例如 500 ms、1 秒或更長，依遊戲狀態調整。
- 不要同時啟動多個屏幕識別、圖片坐標、全局監聽 OCR。
- OCR 結果先寫入變量，再讓後續條件讀變量，避免每個條件都重新識別一次。
- OCR 空值、`ERROR`、超時都要有返回值，不讓主流程停在未知狀態。
- 在模擬器上優先測兼容屏幕檢測模式；仍不穩時改找圖、找色或外部 OCR。
- 子腳本中再調屏幕識別更容易排查困難，核心 OCR 建議集中在主檢測模組。

可用降級順序：

1. 區域 OCR。
2. 區域找圖。
3. 多點找色。
4. 固定 UI 坐標加二次狀態確認。
5. 外部 OCR 或 PC OCR 服務。

## 腳本引擎卡死與停止保護

卡死來源常見於：

- 同時跑多個識別動作。
- 循環中沒有等待。
- 條件符號或比較符號寫錯。
- JS 同步等待網路請求。
- 子腳本互相調用又各自做屏幕識別。
- 大文件下載或大量圖片處理。
- 點擊後不等狀態完成就跳下一步。

穩定腳本應該具備：

- 全局停止旗標。
- 每個狀態的最大等待時間。
- 每個角色或隊伍的失敗次數上限。
- 每輪保存最後狀態。
- 每次異常保存截圖和簡短日誌。
- 長時間運行後自動釋放或重啟識別流程。
- 網路請求使用異步、超時與失敗返回。

對萬龍採集的具體設計：

| 狀態 | 最大等待 | 失敗處理 |
|---|---:|---|
| 角色登入 | 30 秒 | 回角色列表或記錄角色失敗 |
| 回城/出城 | 20 秒 | 用地圖/城內特徵重新判斷 |
| 搜尋資源 | 15 秒 | 換資源等級或種類 |
| 跳轉資源點 | 20 秒 | 重新點搜尋，不盲點採集 |
| 採集按鈕 | 10 秒 | OCR 找不到就保存截圖並換下一資源 |
| 建立部隊 | 30 秒 | 檢查隊伍數、兵力、行軍按鈕 |
| 行軍中 | 60 秒 | 進部隊管理確認狀態 |

## 坐標自適應

分類 3 教程指出，坐標適配不能只靠固定數字。萬龍這類遊戲還要考慮 UI 是否因解析度、比例、城內/野外視角而移位。

建議做法：

- 以基準解析度建立相對坐標，例如 `x / baseWidth`, `y / baseHeight`。
- 關鍵區域用比例框，而不是單點。
- 每個區域都存 `name`, `x1`, `y1`, `x2`, `y2`, `purpose`。
- 用截圖自檢確認放大鏡、回城、採集按鈕、行軍按鈕都在預期框內。
- 同解析度但不同設備仍要重新校準，因為導航列和渲染比例可能不同。
- 能用視覺錨點時，用錨點重新定位，不完全依賴比例坐標。

## 萬龍虛擬測試流程應補的穩定性案例

虛擬測試不只要測「成功採集」，還要刻意餵失敗圖：

- 城內畫面。
- 野外畫面。
- 搜尋面板。
- 找不到資源提示。
- 跳到資源點但沒有採集按鈕。
- 採集按鈕存在但隊伍已滿。
- 部隊管理 0/5、2/5、5/5。
- OCR ERROR 或空白識別結果。
- 模擬器半屏截圖。
- 截圖全黑。
- 腳本停止旗標被觸發。

測試報告應輸出：

- 當前狀態。
- 使用的識別方式。
- 識別結果。
- 點擊位置。
- 下一狀態是否符合預期。
- 失敗時的截圖檔名。
- 角色、隊伍、資源種類與資源等級。

## GitHub 收錄規則

- 保留來源連結，不複製完整原帖。
- 不放破解、繞過、隱藏、反檢測、卡密算法與私有接口。
- 權限問題只整理公開系統設定與相容性排查。
- 模擬器問題整理成兼容矩陣，方便測試而不是保證某一平台一定可用。
- 錯誤案例用「症狀 -> 可能原因 -> 排查步驟 -> 已知限制」格式保存。
