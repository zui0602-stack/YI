## 可转成架构规则

### AI 只当助手，不当最终答案

来自：`25352`, `25318`, `23731`

AI 可用于生成初稿、流程图和表达式，但不能直接替代实测。必须检查：

- 是否使用自動精靈自己的 `zdjl` API。
- 是否把同步/异步 API 混用。
- 是否有网络超时、JSON 解析、空值保护。
- 是否把其它自动化工具的 API 混进来。
- 是否把变量作用域、持久化和运行条件写清楚。

### JS 状态必须显式写回

来自：`18439`, `19879`, `18443`

自動精靈里的 JS 局部变量不会自动成为动作外变量。需要给后续动作或运行条件使用时，应明确写入：

- 当前脚本变量：短流程内使用。
- 全局变量：跨动作、跨子脚本使用。
- 本地存储：跨运行持久保存。

实作建议：

- 判断结果写成布尔值，不要写 `"true"` / `"false"` 字符串再混用反向条件。
- 数组或函数结果不要只存在 JS 局部作用域。
- 常用函数可封装，但这个主题后续优先从分类 3 正式教程和官方 JS 文档整理。

### OCR 结果要先清洗再判断

来自：`23731`, `16379`, `18439`

OCR 输出不是业务数据，必须先清洗：

- 数字比较：去掉非数字字符，再转数值。
- 小数比较：正则要保留小数点和负号的规则。
- 多数字场景：拆成数组后逐个判断，不要只取第一段。
- 答题脚本：题干和选项最好分区识别，不要全屏混扫。

对游戏脚本的意义：

- 资源等级、倒计时、部队数量都应限制识别区域。
- OCR 失败时要有找色/找图兜底，不能让主流程卡死。

### 找图与坐标要以表达式反查为准

来自：`21863`, `20326`

`findLocation` 需要的是自動精靈识别系统能理解的坐标类型对象。稳妥做法：

- 先用界面动作创建一个可工作的坐标变量。
- 长按或查看变量表达式，取得真实对象结构。
- 再把这个结构抽成 JS 模板。
- 所有坐标数组、路径数组使用前先判空。

### 运行环境不是完整 Node.js

来自：`22692`, `21324`, `24388`, `18286`

自動精靈 JS 虽然支持 `require` 和部分 npm 能力，但不是完整桌面 Node.js。需要注意：

- 依赖原生 Node 环境的包不一定能运行。
- 本地包的二级依赖可能仍会走远端 CDN。
- Android 原生类调用可能受线程限制。
- 模拟器 OCR 与真机表现可能不同。

推荐策略：

- 能用内置 `zdjl` API 就优先用内置。
- 大型依赖放到后端或 PC 服务，由脚本走网络请求。
- 关键功能不要依赖不稳定 CDN。

### 后端与托管要区分静态和动态

来自：`21819`, `22209`

GitHub 适合托管静态文件、说明文档、静态配置。需要动态逻辑时要另外设计：

- PHP/MySQL 托管：适合简单登录、卡密、留言、状态接口。
- 自建服务器：适合更复杂的 API、数据库、任务队列。
- 静态 GitHub Pages：不能执行 PHP，只能返回文件内容。

公开 GitHub 时不要放：

- API key
- 卡密生成规则
- 真实后端地址和管理 token
- 用户隐私数据

## 对萬龍覺醒脚本的直接启发

- 角色名、资源文字、倒计时全部做区域 OCR，不做全屏识别。
- OCR 结果先清洗，再写入变量给运行条件使用。
- 找不到资源、没有采集按钮、队伍已满等状态要写成布尔变量。
- 长时间运行要避免依赖远端 npm 包或高频外部 OCR。
- 红手指/模拟器环境要预留 OCR 兼容模式和性能配置排错。
- 后续 GitHub 版本只放架构、模板和脱敏示例，不放卡密或私有接口。
