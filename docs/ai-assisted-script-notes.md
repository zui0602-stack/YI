# AI 輔助腳本專題：Prompt 強約束、知識庫、代碼審核與安全收錄

本文件整理分類 3 教程區與先前已讀分類 4 交流區中與 AI、prompt、知識庫、代碼生成、代碼審核、UI/動作庫相關的內容。分類 4 已於 2026-06-18 停止後續學習，既有內容只保留為歷史參考。

核心結論：AI 適合幫自動精靈腳本拆流程、畫狀態機、生成初稿和整理規格，但不能替代真機/模擬器實測。AI 產出的代碼必須先經過 API、變量、異步、權限、OCR、依賴和安全審核。

## AI 最適合做什麼

適合：

- 把自然語言需求拆成狀態機。
- 把複雜流程拆成動作組。
- 生成變量表、配置表、錯誤狀態表。
- 根據已驗證 API 寫 JS 初稿。
- 根據截圖清單設計虛擬測試案例。
- 幫日誌歸類成問題模式。
- 把論壇問答整理成 GitHub 可讀筆記。

不適合直接相信：

- 未提供 API 文檔時生成的自動精靈代碼。
- 任何使用瀏覽器 DOM、Auto.js、Playwright、普通 Node.js 完整環境的代碼。
- 未判空的坐標、節點、OCR 結果。
- 未設超時的網路請求。
- 未說明權限與環境限制的腳本。
- 直接生成可濫用的驗證、搶購、刷量、風控繞過流程。

## Prompt 強約束模板

給 AI 生成自動精靈代碼前，必須提供這些約束：

- 目標環境是自動精靈 JS，不是瀏覽器，不是完整 Node.js。
- 只能使用已提供的 `zdjl` API 或論壇已驗證 API。
- 禁止使用 `document`、`window`、DOM、瀏覽器事件。
- 涉及坐標、節點、OCR、找圖前，必須先判空。
- 需要後續動作用的結果，必須寫回自動精靈變量。
- 網路請求必須有超時、錯誤返回與 JSON 解析保護。
- OCR 必須限制區域，不能高頻全屏掃。
- 生成代碼後要附「假資料測試案例」和「可能失敗原因」。

萬龍 prompt 應額外加：

- 目前要處理的是遊戲自繪畫面，不能假設有節點。
- 所有坐標必須來自實際截圖或校準資料。
- 每一步都要有狀態確認，不允許盲點下一步。
- 找不到資源、沒有採集按鈕、隊伍已滿都要能回復。
- 輸出狀態機與測試表，比直接輸出一大段點擊代碼更重要。

## AI 代碼審核清單

每段 AI 生成代碼都要檢查：

| 項目 | 檢查問題 |
|---|---|
| 環境 | 是否混入 DOM、瀏覽器、Auto.js、桌面 Node API |
| API | 是否使用真實存在的 `zdjl` API |
| 異步 | `Async` API 是否正確 `await` |
| 變量 | JS 局部變量是否有寫回自動精靈變量 |
| 空值 | 節點、坐標、OCR 結果是否判空 |
| OCR | 是否限制區域、降頻、處理空字串 |
| 坐標 | 是否支持解析度適配或校準 |
| 網路 | 是否有超時、重試、失敗返回 |
| 依賴 | 是否依賴遠端 CDN 或完整 Node 原生模組 |
| 安全 | 是否包含密鑰、卡密算法、繞過驗證或敏感接口 |

## 知識庫該收什麼

來源：`25318`, `15780`

AI 要能穩定幫忙，就不能只丟需求，還要餵「跑過的資料」：

- 官方 JS 文檔與 API 表。
- 已跑通的短代碼片段。
- 真實錯誤日誌。
- 真實截圖與狀態名稱。
- 變量命名規則。
- 已知不能用的 API 或依賴。
- 模擬器/紅手指兼容問題。
- 萬龍各畫面的可識別特徵。

不收：

- 密鑰、token、卡密算法。
- 付費源碼完整實作。
- 可繞過平台安全機制的完整流程。
- 私人後端地址與管理接口。

## AI 圖片理解與 OCR 的分工

來源：`23546`, `23731`

AI 圖片理解適合：

- OCR 很難解析的表格、帳單、截圖結構化資料。
- 需要把圖片中的多欄內容轉成 JSON 或 Markdown。
- 低頻、高價值的資料解析。

OCR 更適合：

- 遊戲倒計時。
- 資源等級。
- 固定區域短文字。
- 高頻狀態判斷。

萬龍結論：

- 採集流程不要把 AI 圖片理解放在高頻循環。
- 找不到資源或異常截圖可以低頻送 AI 做診斷。
- 角色/資源/行軍狀態仍以 OCR、找圖、找色和狀態機為主。

## 常見 AI 錯誤模式

來源：`20111`, `20326`, `18439`, `18443`, `20760`

| 錯誤 | 症狀 | 修正 |
|---|---|---|
| 把自動精靈當瀏覽器 | 出現 DOM、button、document | 改成 `zdjl` API 或自動精靈 UI |
| 變量只在 JS 內有效 | 後續運行條件讀不到 | 顯式寫回自動精靈變量 |
| 坐標未定義 | 讀 `[0]` 時報錯 | 使用前判空與檢查陣列長度 |
| 布爾值被當字串 | 反向條件混亂 | 寫入真正布爾值 |
| URL 變量拼接錯 | 請求地址多出引號或未編碼 | 先取值、編碼，再拼接 |
| OCR 結果直接判斷 | 牛頭不對馬嘴 | 限區、清洗、降級 |

## 代碼庫、UI 庫與 DSL

來源：`18458`, `18050`, `22075`, `22076`, `21920`, `21799`

進階教程展示了用純代碼構建動作、UI、Markdown 表格、Kodex 這類 DSL 的可能性。整理結論：

- 它們適合有 JS 基礎的人維護大型腳本。
- 可以提高 UI 和動作配置的可複用性。
- 但對新手來說抽象層較重，排錯成本高。
- UI 表格、Markdown、動態彈窗在不同版本和平台上可能有渲染限制。
- 放進萬龍正式腳本前，要先用小型 demo 測試。

GitHub 收錄方式：

- 只記錄架構、用法、風險與來源。
- 不搬運完整庫文件或附件。
- 若未取得授權，不收錄完整源碼。

## 依賴與 npm 限制

來源：`22692`, `21324`

自動精靈不是完整 Node.js：

- 依賴原生 Node 環境的包不一定可用。
- OpenCV 類依賴可能需要原生編譯或完整 Node 能力。
- 本地包的二級依賴可能仍去遠端 CDN 拉取。
- 遠端 CDN 失效會讓腳本在運行時報錯。

推薦：

- 關鍵路徑少依賴外部 npm。
- 能用內建 API 就用內建 API。
- 大型能力放後端或 PC 服務，腳本只做請求。
- 依賴失敗要有可讀錯誤與降級流程。

## 對萬龍腳本的 AI 工作流

1. 先把截圖、狀態、按鈕、失敗場景整理成表。
2. 讓 AI 只產生狀態機與測試表。
3. 人工確認每個狀態的識別方式。
4. 再讓 AI 產生小函數，不產生整個大腳本。
5. 每個小函數都用虛擬測試圖跑過。
6. 實機/紅手指再測坐標與性能。
7. 失敗截圖回填知識庫。

萬龍 AI prompt 範例方向：

- 「請根據這 19 張截圖，列出每張屬於哪個狀態。」
- 「請產出找不到資源時的狀態轉移，不要寫坐標。」
- 「請檢查這段代碼是否有未判空、變量未寫回、OCR 高頻問題。」
- 「請把失敗日誌分類成權限、OCR、坐標、流程、資源不足。」

## GitHub 安全規則

- 所有 AI prompt、模板、測試表可以公開。
- 真實 API key、OCR key、後端 token 不公開。
- 第三方模型請求範例要用假 key。
- 付費源碼、商業腳本、附件庫不直接搬。
- 對可能濫用的腳本，只寫安全邊界與抽象原理。
