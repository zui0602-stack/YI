## 核心結論

自動精靈 UI 的核心不是漂亮，而是讓腳本可配置、可觀測、可恢復。正式腳本的 UI 應該分成三層：

- 配置層：使用者輸入資源類型、角色清單、搜尋次數、是否啟用偵錯。
- 狀態層：腳本顯示目前角色、目前狀態、最近錯誤、運行時長。
- 維護層：保存配置、載入配置、重置配置、匯出日誌、查看版本。

不建議把核心流程綁死在 UI 上。UI 可以幫人設定和查看，但採集、行軍、切角色這類核心流程必須能在沒有人工互動時自行跑完。

## 設定變量 UI

### 1. UI 從哪裡來

教程反覆確認：自動精靈內建 UI 主要由「設定變量」動作提供。變量勾選「需要界面手動輸入」後，腳本執行時會形成配置彈窗。

常見變量類型：

- 字符：角色名、配置名、Webhook 類假資料。
- 數字：搜尋次數、資源等級、超時秒數。
- 布爾：是否啟用某功能、是否顯示進階選項。
- 按鈕：切換頁籤、保存配置、重置配置。
- 文本提示：顯示說明、狀態、Markdown。

設計原則：

- 變量名要能直接看懂，不用只寫 `a`、`b`、`x1`。
- UI 只是入口，核心流程讀取配置中心。
- 需要長期保存的設定應寫入 storage 或 JSON。
- 臨時狀態不要放進配置 UI，避免使用者誤改。

### 2. UI 不是資料庫

「記住值」適合簡單配置，例如資源等級或是否啟用偵錯；但多套配置、角色隊列、動態列表應使用 storage 或 JSON。

推薦分工：

| 類型 | 建議保存位置 | 理由 |
|---|---|---|
| 使用者手動選項 | UI 變量 + 記住值 | 易懂，適合少量配置 |
| 多套配置 | storage / JSON | 可切換、可備份 |
| 運行狀態 | state JSON / 腳本變量 | 不污染配置 |
| 錯誤與截圖 | log / 文件 | 方便排查 |

萬龍例子：

- `resourcePlan` 放 JSON。
- `roleList` 放 JSON 或 storage。
- `currentState` 是運行狀態，不放 UI 讓人手改。
- `debugEnabled` 可以放 UI。

## 顯示、隱藏與分頁

### 1. 顯示/隱藏

顯示與隱藏教程的核心是布爾變量。某個區塊是否顯示，取決於一個條件表達式，例如「高級設定是否開啟」。

適合用途：

- 展開/收起進階選項。
- 根據資源類型顯示不同配置。
- 只有開啟多角色時才顯示角色列表。
- 偵錯模式才顯示 OCR 結果與截圖按鈕。

注意：

- 反向顯示常用 `!`，但要確認變量是真布爾，不是字串。
- 顯示條件要保持簡單，複雜判斷先寫成 JS 結果再給 UI。
- 不要讓同一組 UI 同時受多個互相矛盾的條件控制。

### 2. 分頁 UI

分頁 UI 本質是多個布爾頁面狀態：

```text
page_basic = true
page_gather = false
page_roles = false
page_debug = false
```

每個頁籤按鈕做兩件事：

1. 把自己的頁面設為 `true`。
2. 把其他頁面設為 `false`。

萬龍建議頁面：

| 頁面 | 內容 |
|---|---|
| 基本 | 啟用腳本、目標資源、回城/世界地圖策略 |
| 採集 | 資源等級、搜尋次數、找不到資源策略 |
| 角色 | 角色清單、切號順序、失敗跳過 |
| 隊伍 | 最大隊伍、隊伍滿策略、採集/返回判斷 |
| 偵錯 | 當前狀態、OCR 結果、最近錯誤、保存截圖 |

## 單選、多選與動態選項

### 1. 單選按鈕

單選教程提供兩種思路：

- 少量選項：每個按鈕點擊時，自己為真，其他為假。
- 多個選項：用函數或列表統一處理互斥。

對正式腳本，資源類型這種固定選項不一定需要自定義單選 UI，下拉選單或普通選項更簡單。

適合自定義單選的場景：

- 要顯示圖標或特殊文字。
- 要做多選與單選共存。
- 要讓狀態面板更直觀。

風險：

- PC 端和手機端樣式可能不同。
- 選項越多，互斥邏輯越容易漏。
- 按鈕文字用 JS 表達式時要注意變量作用域。

### 2. JS 動態生成選項

交流區問到能否用 JS 生成設定變量選項。整理後的安全答案：

- 簡單選擇可用 `zdjl.select` 類能力。
- 若要嵌入設定變量 UI，需要理解動作源碼或使用動作 JSON。
- 不建議用大量字串拼 JSON；應用物件組裝，再序列化。
- 動態選項要能回寫到變量或 storage。

萬龍應用：

- 角色清單可由 JSON 生成。
- 資源類型固定，不需要動態生成。
- 已偵測到的城堡或帳號列表可作為進階工具，不放核心流程。

## Markdown 與狀態面板

### 1. Markdown 基礎

Markdown 教程說明 `#MD` 可用於 richer 提示，例如標題、換行、顏色、表格、列表與代碼。GitHub 版整理時只保留通用模式，不搬完整格式清單。

適合用途：

- 顯示運行狀態。
- 顯示配置摘要。
- 顯示錯誤報告。
- 顯示多角色進度。
- 顯示採集統計。

### 2. 運行時長與當前時間

舊教程用秒數累加顯示運行時間，回覆建議更穩的做法是：腳本開始時記錄開始時間，需要顯示時用當前時間減開始時間。

推薦模式：

```js
// 原創示意：只保存模式。
function formatRunTime(startAt) {
  const elapsed = Date.now() - startAt;
  const h = Math.floor(elapsed / 3600000);
  const m = Math.floor((elapsed % 3600000) / 60000);
  const s = Math.floor((elapsed % 60000) / 1000);
  return h + '小時' + m + '分' + s + '秒';
}
```

萬龍狀態面板建議顯示：

```text
#MD
### 萬龍採集狀態
- 角色：role-01
- 狀態：SEARCH_RESOURCE
- 資源：gold level 5
- 搜尋次數：2 / 8
- 可用隊伍：3
- 最近錯誤：無
```

### 3. Markdown 表格

Markdown 表格適合展示角色列表、資源隊列與錯誤統計。但交流區也有人遇到表格崩潰或不同平台不顯示，整理成規則：

- 表格先用少量資料測試。
- 行數太多時分頁或分批顯示，不要一次塞數百行。
- 圖片、按鈕、表格混用時要做手機與 PC 雙端測試。
- 出錯時先看自動精靈懸浮窗日誌。
- 設定變量與系統彈窗對 Markdown 支持可能不同。

對萬龍：

- 偵錯面板用簡短列表比大表格更穩。
- 多角色統計可用表格，但要限制行數。
- 失敗報告可寫文件，再只在 UI 顯示摘要。

## 彈窗與生命週期

### 1. 彈窗提示

基礎彈窗可做公告、選擇、前往腳本、退出腳本、坐標設定。正式腳本中彈窗的作用應該清楚：

- 啟動前確認配置。
- 首次使用引導。
- 重大錯誤需要人工處理。
- 偵錯模式下展示資料。

不建議：

- 長時間掛機流程中頻繁彈窗。
- 把每一步狀態都用彈窗打斷。
- 讓核心流程必須等人點彈窗才繼續。

### 2. 動態彈窗重複顯示

交流區確認：動態生成的臨時彈窗即使帶一次性顯示參數，也可能每次都被視為新彈窗。

穩定做法：

```js
// 原創示意：用狀態旗標，而不是依賴臨時彈窗記錄。
const shown = zdjl.getStorage('hasShownConfigDialog') || false;
if (!shown) {
  // run dialog action here
  zdjl.setStorage('hasShownConfigDialog', true);
}
```

### 3. 彈窗 60 秒默認選擇

問答中的思路是：按鈕都設跳轉，彈窗後面的動作也設跳轉；超時後自然走預設分支。整理成架構：

- 選擇 A：跳到 `branch_a`。
- 選擇 B：跳到 `branch_b`。
- 超時未選：彈窗後續動作跳到 `branch_default`。

萬龍中可用於：

- 啟動前 60 秒未選，使用上次配置。
- 重大錯誤倒數後停止腳本。
- 偵錯模式下等待人工確認。

### 4. 彈窗變量與 JS 作用域

多篇問答都指向同一件事：JS 裡的局部變量不會自動給後續動作用。要讓彈窗或後續動作讀到結果，必須顯式寫回自動精靈變量。

常見坑：

- `var result = ...` 後續動作讀不到。
- 彈窗中按鈕改了局部值，但沒有 `setVar`。
- 節點數或 OCR 結果印得出來，但系統彈窗顯示不到。

規則：

- 後續流程要讀：寫回變量。
- 下次啟動還要讀：寫入 storage。
- 只在當前 JS 使用：局部變量即可。

### 5. 系統與鍵盤兼容

有交流帖提到高版本系統中，彈框可能被鍵盤遮擋，回覆指出與手機系統版本相關，後續版本嘗試優化。整理成 GitHub 可用規則：

- 複雜彈窗要在目標設備測試。
- 文字輸入框多時，避免彈窗層級太深。
- 重要配置提供 JSON / storage 備用方式。
- 若鍵盤遮擋，降低 UI 複雜度或拆成多頁。

## 實時監控彈窗

交流區回答很短但重要：全局監聽中，每次運行前關閉彈窗。整理成更完整規則：

- 不要真的讓一個高頻背景流程不停點屏幕。
- 在每個主步驟前後加入 `recover_close_popup`。
- 長等待期間可短間隔檢查安全彈窗。
- 彈窗處理要限制區域和按鈕，避免誤關重要確認。

萬龍彈窗分級：

| 類型 | 處理 |
|---|---|
| 活動廣告 | 可自動關閉 |
| 網路錯誤 | 記錄後重試或重啟 |
| 更新提示 | 停止或進入更新流程 |
| 確認消耗資源 | 不自動確認 |
| 授權/權限 | 優先節點處理 |

## 進階 UI 框架

### 1. ZEasyUI / just4.js

這些庫的價值是用純代碼批量建立 UI、同步變量、減少重複設定。它們適合有 JS 基礎、已經做過腳本的人。

可學架構：

- 把 UI 元件抽成物件。
- 讓配置屬性與自動精靈原生變量保持一致。
- 支持批量修改展示屬性。
- 減少回調與重複點設定。

不建議：

- 在 GitHub 學習庫搬完整庫源碼。
- 在萬龍主流程早期引入大型 UI 抽象。
- 讓 UI 框架失效時整個採集流程不能跑。

### 2. Kodex

Kodex 是運行時 DSL UI 渲染引擎，使用 JSX / Signal 思路構建對話框。教程也提到兩種使用方式：

- 編譯環境：透過 Node/Vite 打包產物。
- 原生環境：載入運行時與 JSX parser。

整理結論：

- Kodex 適合複雜互動 UI。
- 它比原生設定變量重，依賴包與網路情況會影響穩定性。
- 交流回覆出現過 CDN 取得不到的錯誤，因此正式腳本要有降級方案。
- 對萬龍採集來說，Kodex 更適合做配置工具，不適合做核心採集循環依賴。

### 3. HTML-UI

HTML-UI 思路是用 HTML 表單收集配置，再把結果傳回自動精靈。它可用於複雜表單，但會引入瀏覽器頁面、子腳本參數、資料傳遞問題。

適合：

- 一次性配置向導。
- 大量文字輸入。
- 複雜表單排版。

不適合：

- 高頻狀態面板。
- 核心掛機流程。
- 需要無人值守的每輪判斷。

## 長時間 UI 與內存

播放長時間視頻的 UI 教程不是萬龍必需，但它提供了重要的性能觀念：自動精靈 JS 環境內存有限，不能把大量圖片或資料一次塞進 storage 或內存。

可學原則：

- 大資料用流式載入。
- 固定大小滑動窗口。
- 播放過或用過的資料要釋放。
- UI 動畫、圖片、表格會影響後續腳本性能。

萬龍應用：

- 不要一次把大量截圖 base64 存進 storage。
- 失敗截圖按需保存文件，不塞 UI。
- UI 只顯示摘要，詳情放日誌。

## 萬龍覺醒配置界面建議

### 1. 最小 UI

第一版建議只做必要項：

| 欄位 | 類型 | 預設 |
|---|---|---|
| 啟用採集 | 布爾 | true |
| 資源類型 | 選項 | gold |
| 最低等級 | 數字 | 5 |
| 最高搜尋次數 | 數字 | 8 |
| 多角色 | 布爾 | false |
| 偵錯模式 | 布爾 | true |

### 2. 狀態面板

狀態面板用 Markdown 短列表：

```text
#MD
### 狀態
- 角色：role-01
- 階段：SEARCH_RESOURCE
- 資源：gold Lv5
- 搜尋：2/8
- 錯誤：none
```

### 3. 進階配置

進階頁面才放：

- 多角色列表。
- 隊伍策略。
- 超時設定。
- OCR 區域。
- 截圖保存。
- 更新檢查。

### 4. 禁止把這些放 UI 讓人手改

- `currentState`
- `lastAction`
- `lastDetection`
- `failureCount` 的內部累計
- 未脫敏 token 或後端地址
- 付費源碼或完整附件內容

## 虛擬測試案例

UI 也要測，不只測採集流程。

| 測試 | 期望 |
|---|---|
| 第一次啟動 | 顯示配置，保存後寫入 storage |
| 第二次啟動 | 使用已保存配置，不重複彈出引導 |
| 切換資源類型 | 對應配置顯示，其他資源隱藏 |
| 切換分頁 | 只有當前頁內容顯示 |
| 開啟偵錯 | 顯示狀態面板 |
| 關閉偵錯 | 核心流程仍能執行 |
| Markdown 表格 5 行 | 正常顯示 |
| Markdown 表格 100 行 | 不崩潰；若慢，改分頁 |
| 彈窗超時 | 走預設分支 |
| 鍵盤遮擋 | 有替代輸入或拆頁 |

## 代碼審核清單

| 項目 | 檢查問題 |
|---|---|
| UI 作用 | 是否只做配置/顯示，不阻塞核心流程 |
| 變量 | 需要後續讀取的值是否 `setVar` 或 storage |
| 顯示條件 | 是否基於清楚布爾值 |
| 分頁 | 是否只有一頁為真 |
| 單選 | 是否互斥完整 |
| Markdown | 是否控制資料量 |
| 彈窗 | 是否有超時或預設分支 |
| 動態 UI | 是否使用狀態旗標避免重複顯示 |
| 兼容 | 是否測過目標手機/雲機/PC |
| 依賴 | 是否依賴 CDN、第三方 UI 庫、瀏覽器 |
| 安全 | 是否避免公開 token、後端地址、付費源碼 |

## GitHub 收錄規則

- 保留來源連結與 Thread ID。
- 不搬運 ZEasyUI、Kodex、just4.js 或 HTML-UI 附件完整源碼。
- Markdown 範例只放短模板，不搬整篇格式手冊。
- 外部影片只記錄學習主題與用途，不下載到倉庫。
- 進階 UI 庫只整理架構、風險和適用場景。
- 萬龍配置範例只放假資料。
