## 识别方式选择

| 方式 | 适合 | 不适合 |
|---|---|---|
| 节点 | 普通 App、系统弹窗、权限页 | 游戏自绘画面 |
| OCR | 角色名、倒计时、数字、提示文字 | 背景复杂、字体花、单字符 |
| 找图 | 固定图标、按钮、资源图 | 图标缩放、光照变化、动态图 |
| 找色 | 颜色稳定的按钮、状态点、进度条 | 颜色受光照/特效影响 |
| 多点多色 | 固定 UI 图案、多个相对色点 | 动画变化、分辨率差异很大 |
| 区域颜色 | 血条、进度条、冷却条 | 颜色渐变复杂或透明度高 |
| 外部 OCR | 本地 OCR 不准、需要更强模型 | 高频循环、网络不稳、隐私敏感 |
| PC OCR | 同局域网高频识别 | 外网复杂、端口/防火墙不稳定 |

## OCR 数字比较标准流程

来自：`13534`, `6926`, `16379`

标准链路：

1. 设置变量，变量类型选择屏幕识别。
2. 限制识别区域，只框数字或目标文字附近。
3. 用 JS 清洗结果，例如去掉非数字字符。
4. 转成数值。
5. 将判断结果写入变量。
6. 后续动作用运行条件判断。

关键原则：

- OCR 得到的通常是字符串，不要直接当数字用。
- 如果识别到 `+22649`、空格、换行、单位，要先清洗。
- 多个数字要拆数组逐个判断，不要只取第一段。
- 阈值应该做成变量，方便 UI 设置。

对万龙觉醒：

- 部队数、资源等级、倒计时都应独立区域识别。
- 不要全屏扫角色名，会误识别地图上的城名、联盟名和聊天文字。

## 单字符识别问题

来自：`22411`

单个字符常常识别率差，解决思路：

- 尽量扩大识别上下文，让单字符变成词或短句。
- 对白底黑字等简单场景可预处理图像。
- 用找图/找色替代单字符 OCR。
- 对结果做模糊匹配，而不是要求完全相等。

## 外部 OCR

来自：`25439`, `8099`, `25093`

外部 OCR 路线：

- OCR.space：申请 API key 后用 multipart 上传截图。
- 百度 OCR：需要 AK/SK 换取 access token，token 有有效期。
- PC Paddle/Umi OCR：在 PC 启服务，手机通过局域网或公网访问。

实践规则：

- API key、AK/SK 不放 GitHub。
- 截图只截必要区域，降低成本和误识别。
- 外部 OCR 必须有超时、重试、失败降级。
- token 过期要自动刷新或从远端配置拉取。
- PC OCR 要处理 IP、端口、防火墙和小图识别能力。

## 找图

来自：`24122`, `10679`, `21863`

找图参数重点：

- 限制区域不要太大，也不要太小，通常以目标附近 2-5 倍区域为起点。
- 相似度要实测，不是越高越好。
- 快速搜图可降低资源占用，但可能影响精度。
- 搜图模式改变后要重新调相似度。
- 多分辨率适配要单独测试。

全屏随机位置目标：

- 普通找图全屏扫容易慢和误判。
- 动态分割识图思路是把屏幕切块，逐块识别，失败后扩大块或调整相似度。
- 可加入历史成功位置记忆，下次优先搜索高概率区域。

## 多点多色与位置周围条件

来自：`16545`, `15692`, `14761`

核心思想：

- 单点颜色容易撞色。
- 先找基准点，再检查基准点附近多个相对位置颜色。
- 多个点全部满足时才算目标出现。

使用建议：

- 基准点选择稳定且不易被遮挡的位置。
- 相似度不要全部设 100，实际应用常要留容错。
- 点位数量越多越准，但越慢。
- 动图或会变色目标不适合固定多点多色。
- 识别范围太大或点太多可能卡死。

对万龙觉醒：

- 固定 UI 按钮可用多点多色。
- 资源图标如果有光照/阴影变化，找色要谨慎。
- “采集”按钮可以先用 OCR/找图，失败再用按钮颜色范围兜底。

## 区域颜色与进度条

来自：`18438`

进度条不一定要 OCR 倒计时。可以用区域颜色数据换算比例：

- 直线进度条：统计目标颜色占比。
- 圆形进度条：按圆周点或区域颜色采样估算。
- 血条/冷却条：颜色稳定时比 OCR 更快。

注意：

- 透明、渐变、光效会影响颜色判断。
- 进度条会移动时要先定位条的位置，再取区域颜色。
- 高频循环里区域颜色通常比 OCR 和找图更适合。

## 特征匹配与 ORB

来自：`22160`

ORB 特征匹配适合普通找图难以处理的情况，例如旋转、光照变化。已读教程中的关键限制：

- 对对比度敏感。
- 低对比度噪声多。
- 太大的目标图可能被压缩，影响匹配。
- 自动精灵电脑版和移动端返回数据类型可能不同。
- 不一定支持完整多尺度匹配。

结论：

- ORB 是高级备选方案，不应作为第一层识别。
- 优先顺序仍是：区域限制 -> 找图/找色/OCR -> 多点多色 -> 特征匹配。

## 多目标距离判断

来自：`4581`

常见游戏问题：屏幕上有多个怪/资源点/按钮，想点最近一个。

模式：

1. 找出人物坐标。
2. 找出所有目标坐标。
3. 对每个目标计算距离。
4. 选择距离最小的目标。
5. 点击该目标。

距离可以用：

- 欧式距离：适合真实屏幕距离。
- 曼哈顿距离：计算简单。
- 横向/纵向优先：适合地图移动或横版游戏。

对万龙觉醒：

- 如果同时出现多个资源点，可按距离或资源等级排序。
- 若目标坐标来自 OCR 或找图，要先过滤误识别点。

## 点击策略

来自：`24122`

点击优先级：

1. 节点点击：普通 App 最稳。
2. 图片点击：图标和按钮较稳。
3. OCR 文字点击：文字按钮可用，但需限制区域。
4. 颜色点击：适合颜色稳定的 UI。
5. 坐标点击：最后兜底，必须做好分辨率适配。
6. AI 点击：可尝试，但弹窗层级和限制区域要特别注意。

游戏脚本原则：

- 游戏自绘界面通常没有稳定节点，优先视觉识别。
- 坐标点击必须用百分比或按分辨率换算。
- 每次点击后要检查下一状态，不能只相信点击已成功。
