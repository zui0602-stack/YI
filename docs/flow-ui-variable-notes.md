## 核心架構

大型腳本不要只靠「跳轉 +1」堆流程。比較穩的做法是把腳本拆成三層：

- 任務層：啟動遊戲、回城、找資源、派隊、切角色、收尾。
- 判斷層：畫面是否正確、是否找到資源、是否出現採集、隊伍是否空閒、角色是否登入成功。
- 控制層：循環、超時、失敗重試、動作鎖、狀態紀錄。

教程中反覆出現的共同思路是：把可重用的動作封成變量或動作組，再由 JS 或監聽流程負責調度。這樣 UI 點擊、OCR、找色等底層能力可以保持單純，整體流程也比較容易讓 AI 或人類檢查。

## 流程控制

### 1. 變量式任務循環

任務循環教程的重點是：把一段任務動作包成「變量值為動作組」的變量，任務最後再依條件調用自己或下一個任務。這比單純跳轉更適合多帳號、多角色和大型腳本，因為任務可以被其他流程重複調用。

建議規則：

- 每個任務都有明確入口，例如 `task_gather_gold`、`task_switch_role`。
- 每個任務都有明確成功條件，例如「進入採集面板」、「隊伍狀態變為行軍」。
- 每個任務都有失敗出口，例如「回到主城」、「重啟遊戲」、「切下一角色」。
- 任務內部可循環，但必須有最大次數或超時。

### 2. 監聽動作與生命週期

監聽流程分成全局監聽與動作監聽。全局監聽適合處理腳本級事件，例如啟動前初始化、運行前檢查、超時停止；動作監聽適合處理單個動作的成功、失敗和結束後清理。

可整理成這個執行順序：

1. 腳本啟動後初始化全局變量與配置。
2. 動作運行前檢查必要狀態。
3. 運行條件滿足後才執行實際動作。
4. 成功後更新狀態、時間戳、計數或下一任務。
5. 失敗後跳到恢復流程，而不是讓後續動作盲目繼續。
6. 結束後清理臨時變量或記錄日誌。

### 3. 失敗回復

交流區的問答給出的簡化答案是：前一個動作失敗時，用動作監聽的「運行失敗後」跳到指定恢復位置。整理成架構後，建議不要每個動作各自亂跳，而是集中到幾個恢復入口：

- `recover_close_popup`：關閉彈窗。
- `recover_back_home`：回到主城或世界地圖安全畫面。
- `recover_restart_game`：重啟遊戲。
- `recover_next_role`：本角色失敗過多，切下一個角色。

### 4. 動作運行鎖

運行鎖適合「某動作不能每輪都跑」的情境。例如技能循環、資源搜尋頻率、切角色冷卻。基本模式是：

- `locked`：布爾值，表示是否允許執行。
- `count`：累計其他動作成功次數。
- `limit`：達到幾次後解鎖。
- 目標動作成功後重新上鎖並清空計數。

這種設計比到處放時間等待更清楚，尤其適合長時間掛機腳本。

### 5. 定時與多任務輪詢

「動作完成後延遲執行」與「多個固定時間循環」的共同模式是記錄下一次可執行時間：

```js
// 這是整理後的偽代碼，不是搬運原帖完整代碼。
nextRunA = Date.now() + fixedDelayMs;
nextRunB = Date.now() + dynamicDelayMs;

if (Date.now() >= nextRunA) runTaskA();
if (Date.now() >= nextRunB) runTaskB();
```

用在遊戲腳本時，重點是「輪詢條件」而不是長時間阻塞等待。等待可以短，例如每 1 到 3 秒檢查一次，避免卡住其他任務。

### 6. 單次循環超時

交流區提到可用全局監聽或子腳本控制單次運行時長。整理成萬龍採集架構時，應該每個角色、每個資源流程都設置開始時間與最大時長：

- 角色採集流程開始時記錄 `roleStartAt`。
- 找資源階段記錄 `searchStartAt`。
- 派隊階段記錄 `dispatchStartAt`。
- 任一階段超過上限就進入恢復流程。

## 變量與存儲

### 1. 變量作用分層

變量不要全部混在一起。建議分成：

- UI 配置變量：資源種類、等級、角色列表、是否啟用某功能。
- 狀態變量：目前角色、目前任務、當前階段、失敗次數。
- 結果變量：OCR 結果、找圖坐標、是否找到採集按鈕。
- 持久化變量：下次啟動仍要保留的配置。

### 2. 記住值與本地存儲

坐標變量若只要下次運行還能使用，可在變量界面使用「記住值」。但若要由 JS 自動更新並長期保存，交流區建議改用本地存儲。整理後原則如下：

- 使用者手動設定的固定坐標：可用「記住值」。
- 腳本自動更新的配置或列表：用 `zdjl.setStorage` / `zdjl.getStorage` 類型的持久化方式。
- 臨時狀態：只放腳本變量，不要持久化。

### 3. 布爾值不要寫成字串

時間段判斷的問答提醒了一個常見坑：如果運行條件要讀布爾值，盡量存真正的布爾值，不要存 `"true"` / `"false"` 字串。否則條件反向、比較方式或 UI 顯示容易混亂。

### 4. 坐標是否已設定

坐標變量未設定時，可以用「讀取變量後是否為空」來判斷。整理成可維護規則：

- 坐標不存在：進入設定流程或提示使用者選點。
- 坐標存在：直接執行點擊或找圖區域。
- 坐標點擊失敗：不要立刻覆蓋舊值，先提示或進入重新校準流程。

## UI 設計

### 1. UI 來源

自動精靈的 UI 多由「設置變量」提供。變量勾選需要界面手動輸入後，就能形成設定彈窗。教程中 UI 不只是美觀，核心價值是讓腳本可配置、可檢查、可復用。

### 2. 顯示與隱藏

顯示/隱藏教程的關鍵是用布爾變量控制某些 UI 項是否出現。常見用途：

- 高級選項預設隱藏。
- 選擇資源類型後，只顯示該資源相關配置。
- 選擇「多角色」後，才顯示角色列表與切號設定。

### 3. 分頁 UI

分頁 UI 的本質是多組布爾值：目前頁面為真，其他頁面為假。每個頁籤按鈕負責切換這些布爾變量，內容區再根據變量決定顯示。

萬龍腳本建議分頁：

- 基本：啟動、回城、資源種類。
- 採集：資源等級、搜尋次數、採集滿載策略。
- 角色：角色清單、登入順序、失敗跳過。
- 偵錯：OCR 結果、找色結果、當前階段、最後錯誤。

### 4. 單選按鈕

單選按鈕的做法是：選中一個時讓其他選項為假。選項少時可用表達式，選項多時應封成函數或統一切換動作。這對資源類型選擇很適合，例如金礦、伐木場、鐵礦、魔水。

### 5. 狀態提示面板

局數、運行時長、當前時間、顏色狀態提示的教程可用來做腳本內狀態面板。萬龍腳本建議顯示：

- 目前角色。
- 目前任務階段。
- 搜尋資源類型與等級。
- 最近一次 OCR 或找色結果。
- 連續失敗次數。
- 本輪已運行時間。

## 動態 UI 與限制

動態生成彈窗的問答指出：臨時生成的彈窗每次都可能被視為新彈窗，`dialogShowOnce` 不一定能阻止重複顯示。穩定做法是自己加一個持久或腳本級旗標，例如 `hasShownConfigDialog`，顯示前先檢查，使用者確認後再設為真。

對 GitHub 學習庫的整理原則是：動態 UI 可以學架構，但不要把論壇中的完整生成字串整段搬過來。實作時用明確 JSON 物件或封裝函數建立，避免大量字串拼接難以維護。

## 動作源碼與 JS 調用

查看動作表達式和源碼的教程很重要。學到的架構是：

- 先在自動精靈 UI 裡做好一個小動作。
- 查看或導出該動作表達式。
- 把動作抽象成可重用函數或變量。
- JS 只負責調度，不要把所有圖色與 UI 細節寫死在一大段代碼裡。

這也解決交流區「JS 能否調用已編輯動作」的問題：可透過動作源碼、動作變量或 `runAction` 類模式，把 UI 裡做好的動作納入 JS 調度。

## 萬龍覺醒採集腳本應用

### 建議狀態機

```text
INIT
  -> LOGIN_ROLE
  -> ENSURE_SAFE_SCREEN
  -> OPEN_SEARCH
  -> SELECT_RESOURCE
  -> SEARCH_RESOURCE
  -> CHECK_RESOURCE_FOUND
  -> OPEN_RESOURCE_PANEL
  -> CHECK_GATHER_BUTTON
  -> CREATE_OR_SELECT_TROOP
  -> DISPATCH
  -> VERIFY_MARCHING
  -> NEXT_ROLE_OR_WAIT
  -> RECOVER on failure
```

### 每個狀態要有三件事

- 進入條件：目前畫面、角色、隊伍狀態是否符合。
- 成功出口：下一個狀態。
- 失敗出口：重試、回城、重開、切角色或停止。

### 虛擬測試要覆蓋的畫面

這份專題對虛擬測試最有用的地方是把每個畫面變成狀態測試：

- 搜尋資源面板：資源種類、等級、搜尋按鈕。
- 找不到資源文字：不能跳到採集，應改資源或重試。
- 資源詳情有採集按鈕：進入派隊。
- 採集按鈕不存在：進入恢復或改目標。
- 部隊管理已有採集隊伍：判斷剩餘隊伍數。
- 角色管理與登入確認：切角色流程要有確認與超時。
- 世界/城市切換：放大鏡與回城按鈕要獨立校準。

## 實作守則

- 每個循環都要有退出條件。
- 每個長流程都要有超時。
- 每個 OCR/找色結果都要寫入狀態提示，方便截圖排錯。
- 每個角色都要有獨立失敗次數，避免單一角色卡死整個腳本。
- 每個設定都要分清楚「臨時狀態」和「下次仍需保留」。
- UI 只負責設定與顯示，核心流程要能在沒有人工互動時跑完。
- 動態生成內容要有自己的顯示旗標，不能完全依賴一次性彈窗參數。
