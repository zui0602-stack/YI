## 什麼時候用節點

適合節點的場景：

- 普通 App 頁面，例如微信、表單、列表、瀏覽器、系統權限頁。
- 有可讀 `text`、`desc`、`id`、`className`、`packageName`、`depth` 的控件。
- 文字會變，但節點結構穩定。
- 需要輸入框、按鈕、列表項、系統彈窗的精準操作。

不適合節點的場景：

- 遊戲主畫面、地圖、戰鬥、資源點，多半是自繪。
- WebView 內部或 canvas 畫面，控件樹可能只有外殼。
- 圖片、動畫、特效、進度條，通常沒有可靠文字節點。
- 節點每次刷新後結構完全不同，且無法用關係定位。

萬龍覺醒結論：

- 城內、野外、採集、資源搜尋、行軍按鈕多半不要期待節點。
- 系統權限彈窗、截圖授權、紅手指外層 UI、登入器外層頁面可以先查節點。

## 查找節點的基本策略

從穩到不穩的選擇：

1. `id` 或 `idResName` 有明確語義。
2. `text` 精準匹配。
3. `desc` 或描述文字匹配。
4. `className + text/desc` 組合。
5. `packageName + depth + className` 組合。
6. 正則匹配動態文字。
7. 找到父節點或兄弟節點後，再取子節點。

不建議：

- 只靠 `depth`。
- 只靠節點序號。
- 使用看起來像亂碼或每次變動的 id。
- 在列表中用固定坐標假裝是節點定位。

實作提醒：

- 一個選擇器先要求能穩定找到一個目標。
- 找到多個時，要再加條件或用列表過濾。
- 找不到時先重新 dump/刷新節點，不要馬上退到坐標。

## 正則查找節點

來源：`11749`, `17906`, `25334`

節點文字帶數字、日期、倒計時、頁數時，正則比硬編文字穩。典型用途：

- `12/12` 這類前後數字相等。
- 包含日期、金額、數量的動態文字。
- 聊天列表、消息列表、表格列。

整理規則：

- 正則要放在節點條件支持的欄位裡，不要把 JS 的 `RegExp` 物件亂塞到純文字欄位。
- 如果界面支持「變量輸入模式」或正則開關，優先走界面生成的正確表達式。
- 正則只負責找候選，找到後仍要檢查節點屬性或所在區域。

## 節點快取與刷新

來源：`15518`, `17039`, `18188`, `20419`, `20710`

症狀：

- 同一段 JS 多次取節點，`text` 沒刷新。
- 編輯時看得到節點，運行時取不到。
- 微信或列表頁一段時間後節點失效。
- 不同手機同一 App 的節點結果不同。

已整理做法：

- 短時間重複讀取時，關閉節點資料快取可能有幫助。
- 若間隔已經較長仍不刷新，要考慮節點層級或頁面實際結構變了。
- 對微信、文章頁、列表頁這類動態內容，不要把一次截到的節點當永久穩定。
- 查不到節點時保留降級方案：OCR 原始資料、找圖或人工校準區域。

萬龍測試中的用法：

- 如果腳本需要處理系統彈窗，可先查節點。
- 若彈窗節點不穩，才改用 OCR/找色判斷按鈕。
- 每次自檢要記錄「節點可用/不可用」，不要在遊戲流程中臨時猜。

## 節點文字、描述與子節點

來源：`17232`, `17475`, `19542`, `22756`

常見取值模式：

- 文字在 `text`：直接讀 `節點變量.text`。
- 文字在描述：讀描述欄位，視實際節點資料名稱而定。
- 文字在子節點：先取 `children`，再讀子節點文字。
- 文字包含固定分隔符：先切割再取需要的段落。

風險：

- `text` 可能為空，但 `desc` 有值。
- `children[0]` 前要確認 children 存在且長度足夠。
- 分隔符不一定永遠存在，切割後要檢查陣列長度。

整理成通用規則：

- 任何節點資料使用前都先判空。
- 對外輸出時寫入自動精靈變量，不只留在 JS 局部變量。
- 提取結果需要給後續動作用，就用 `setVar` 一類方式顯式寫回。

## 剪貼板、輸入框與批量輸入

來源：`19813`, `17463`, `16567`, `25334`

常見流程：

1. 從節點、OCR 或文件取文字。
2. 清洗文字。
3. 寫入剪貼板或輸入框。
4. 記錄目前處理到第幾行。
5. 下一輪讀下一行。

可複用模式：

- 節點文字複製到剪貼板：取出 `text` 後寫入剪貼板。
- 文件多行輸入：讀文件、按換行切陣列、用索引變量記住進度。
- OCR 原始資料：必要時可讀剪貼板中的 JSON，再解析文字與坐標。

注意：

- 大量輸入要保存索引，避免腳本重啟後從頭輸入。
- 輸入框優先用節點輸入；節點不可用再用剪貼板貼上。
- 剪貼板可能被系統或其他 App 覆蓋，重要資料要同步保存在本地狀態。

## 系統權限彈窗與普通彈窗

來源：`18308`, `25261`, `22917`, `24276`

系統權限彈窗通常是普通 App 控件，應優先嘗試節點：

- 查找「允許」「確定」「取消」「同意」等按鈕文字。
- 若文字因系統語言不同而改變，建立多語言候選。
- 若節點不可用，再用 OCR 限定彈窗區域。
- 最後才用固定坐標。

腳本內自定義彈窗：

- 需要等待使用者選擇時，用跳轉動作設計分支。
- 若需要 60 秒無選擇後默認某項，應把超時流程也設成明確跳轉。
- 彈窗內 JS 局部變量不能自動給後續動作用，要顯式寫回自動精靈變量。
- 部分系統版本可能出現彈框被鍵盤遮擋，應記錄機型與系統版本。

## 列表滑動與避免重複點擊

來源：`21197`

列表自動化常見問題是節點高度不一、滑動後項目重複、點過的項目又出現。

推薦做法：

- 優先使用列表項的唯一文字、id、時間、標題或組合鍵。
- 建立 `visited` 狀態，點過的項目寫入本地變量或 JSON。
- 滑動前後都抓一次可見項目，判斷是否有新項目。
- 若節點不可靠，改用 OCR 原始資料的文字與坐標範圍。
- 每次滑動後加等待，不要立刻讀節點。

## 廣告、驗證與安全邊界

已讀文章中有廣告節點、驗證碼節點、微信紅包等案例。GitHub 學習庫只整理節點定位、資料提取、彈窗處理和穩定性原理，不做下列內容：

- 繞過驗證碼或平台安全機制。
- 幫第三方 App 規避風控。
- 自動搶紅包、搶購、刷量等可濫用流程的完整腳本。
- 廣告關閉的通用對抗手冊。

可公開保留的是：

- 如何判斷節點是否存在。
- 如何從節點取文字。
- 如何處理節點不刷新。
- 如何設計降級與日誌。

## 對萬龍覺醒腳本的實用結論

| 場景 | 節點是否優先 | 建議 |
|---|---|---|
| 遊戲主畫面 | 否 | 用 OCR、找圖、找色、坐標與截圖自檢 |
| 系統截圖授權 | 是 | 先找「允許/立即開始」節點 |
| 無障礙權限頁 | 是 | 先查系統節點，再人工確認 |
| 紅手指外層 UI | 視情況 | 可先查節點，失敗再 OCR |
| 角色管理彈窗 | 多半否 | 遊戲內彈窗通常視覺識別 |
| 腳本設定 UI | 是 | 自動精靈自己的 UI 用變量與彈窗狀態管理 |

萬龍虛擬測試應新增：

- 權限彈窗節點樣本。
- 截圖授權彈窗樣本。
- 節點不可用時的 OCR 降級測試。
- 同解析度但節點/坐標不同的設備記錄。
