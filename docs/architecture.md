# 自動精靈架構理解

## 核心模型

自動精靈可以理解成三層：

1. 視覺/狀態識別層
2. 流程控制層
3. 動作執行層

腳本穩不穩，主要取決於這三層是否分開設計。不要把「看到什麼」「要不要做」「點哪裡」混在同一個積木裡，否則流程一長就很難查錯。

## 視覺/狀態識別層

常見來源：

- 圖片識別
- 文字 OCR
- 節點查找
- 顏色/區域顏色
- 手動設定的座標或區域變量

遊戲腳本多半是自繪畫面，所以節點查找不一定有用；這時優先用 OCR、找圖、找色、固定區域判斷。

## 流程控制層

常見控制方式：

- 運行條件
- JS 表達式
- 變量
- 跳轉動作
- 子腳本/分段腳本
- 循環與計數器

推薦架構：

- 主控腳本只負責調度。
- 每個功能做成獨立段落，例如登入、領資源、採集、切角色。
- 每段都有成功出口、失敗出口、重試上限。
- 重要狀態寫入變量，方便顯示和除錯。

## 動作執行層

常見動作：

- 點擊
- 滑動
- 返回
- 輸入
- 執行其他腳本
- 運行 JS 代碼
- 彈窗/Toast 提示

穩定做法：

- 能用狀態判斷就不要硬等固定秒數。
- 每個點擊後最好有下一畫面確認。
- 會失敗的地方要有「找不到時怎麼辦」。

## 變量作用域

自動精靈變量大致要分成：

- 當前腳本變量：適合單段流程內使用。
- 全局變量：適合跨腳本、跨段落共享，例如角色索引、角色列表、採集隊列。

設計原則：

- 跨 V1-A/V1-B/V1-C 這種模組讀寫的資料，用全局或明確傳遞。
- 單段內暫時計算結果，用當前腳本作用域。
- 不要把所有東西都設成全局，否則舊值會污染下一輪。

## 遊戲腳本建議狀態機

以萬龍覺醒採集為例：

```text
初始化
  -> 選角色
  -> 登入確認
  -> 主城完成
  -> 出城
  -> 打開搜尋
  -> 選資源/等級
  -> 搜尋資源
  -> 找到資源點
  -> 點採集
  -> 創建部隊
  -> 行軍
  -> 檢查部隊數
  -> 未滿繼續 / 已滿換角色
```

每一步都應有「目前畫面是否正確」的判斷，尤其是 OCR 角色名時，不可以在世界地圖全螢幕亂掃，會把地圖上的城堡名誤認成角色列表。
